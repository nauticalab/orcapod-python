# ENG-340 + ENG-349: Decouple Nodes from pipeline_path via ScopedDatabase & Observer Storage Path Migration

**Date:** 2026-04-01
**Issues:** ENG-340 (Refactor: decouple nodes from pipeline_path via ScopedDatabase), ENG-349 (Observer/Logger: migrate storage path scoping)
**Branch target:** `dev`

---

## Overview

This refactor removes `pipeline_path` from the node and observer APIs by pre-scoping databases before passing them in. Nodes and observers no longer receive a path prefix — they receive databases already rooted at the correct location. The combined scope covers ENG-340 (node decoupling) and ENG-349 (observer storage path migration), since both share the same mechanism and it is cleaner to do them together.

No backward compatibility is maintained for previously saved pipeline formats. This is still greenfield.

---

## Section 1: Database Scoping (Pipeline-level)

`Pipeline.compile()` creates a hierarchy of pre-scoped databases from the user-provided root database:

```
pipeline_db  = user_db.at(*pipeline_name_parts)
result_db    = pipeline_db.at("_result")
status_db    = pipeline_db.at("_status")
log_db       = pipeline_db.at("_log")
```

Each derived database is passed directly to the subsystem that needs it:
- `result_db` → `FunctionNode.attach_databases(pipeline_database, result_database)`
- `status_db` → `StatusObserver(status_db)`
- `log_db` → `LoggingObserver(log_db)`

Nodes and observers write paths relative to their pre-scoped database root. The pipeline name and type partition are invisible to them.

### Pipeline properties

`Pipeline` exposes read-only properties:
- `scoped_pipeline_database` — the `pipeline_db` scoped view
- `result_database` — the `_result`-scoped view
- `status_database` — the `_status`-scoped view
- `log_database` — the `_log`-scoped view

## Section 2: Node Interface Changes

### FunctionNode

- Remove `_pipeline_path_prefix: tuple[str, ...]` field
- Remove `_stored_pipeline_path` and all backward-compat prefix-stripping logic
- Rename `pipeline_path` property → `node_identity_path`; returns `pf.uri + (f"schema:{pipeline_hash}", f"instance:{content_hash}")` relative to the pre-scoped database
- `attach_databases(pipeline_database, result_database=None)` — `pipeline_path_prefix` parameter removed entirely
- `execute()` uses `observer.contextualize(*self.node_identity_path)` to bind node identity before calling hooks

### OperatorNode

- Same changes: remove `_pipeline_path_prefix`, rename `pipeline_path` → `node_identity_path`
- `attach_databases(pipeline_database, cache_mode=CacheMode.OFF)` — no prefix parameter
- `from_descriptor()` removes all `pipeline_path_prefix` handling

### CachedFunctionPod

- Remove `record_path_prefix: tuple[str, ...] = ()` parameter from `__init__`
- `self._cache = ResultCache(result_database=result_database, record_path=self.uri)` — path is just `self.uri`, no prefix, since `result_database` is already pre-scoped

## Section 3: Observer Protocol

`ExecutionObserverProtocol` changes:

| Before | After |
|--------|-------|
| `contextualize(node_hash, node_label)` | `contextualize(*identity_path: str)` |
| `on_node_start(label, hash, pipeline_path=(), tag_schema=None)` | `on_node_start(label, hash, tag_schema=None)` |
| `on_node_end(label, hash, pipeline_path=())` | `on_node_end(label, hash)` |
| `create_packet_logger(tag, packet, pipeline_path=())` | `create_packet_logger(tag, packet)` |

`contextualize(*identity_path)` returns a context-bound observer (or self). The identity path flows from `node_identity_path` on the node. All storage path resolution is encapsulated inside the observer implementation — callers never compute paths.

`NoOpObserver.contextualize(*identity_path)` returns `self` (no state needed).

## Section 4: Observer Implementations

### StatusObserver

```python
class StatusObserver:
    def __init__(self, status_database: ArrowDatabaseProtocol) -> None: ...
    def contextualize(self, *identity_path: str) -> _ContextualizedStatusObserver: ...
    def to_config(self) -> dict: ...
    @classmethod
    def from_config(cls, config: dict) -> StatusObserver: ...

class _ContextualizedStatusObserver:
    # Bound to identity_path; writes to status_db.add_record(identity_path, ...)
```

- Receives pre-scoped `status_database` — no path mirroring logic
- `contextualize(*identity_path)` returns a lightweight wrapper with the path bound
- `_node_context` dict removed; context is held in the wrapper, not mutated state
- `get_status()` loses the `pipeline_path` parameter

### LoggingObserver

Same pattern:
- Pre-scoped `log_database`
- `contextualize(*identity_path)` → `_ContextualizedLoggingObserver`
- `PacketLogger` receives `log_path=identity_path` directly
- `get_logs()` loses `pipeline_path` parameter

### CompositeObserver

`contextualize(*identity_path)` delegates to each child and wraps results in a new `CompositeObserver`.

## Section 5: Pipeline API and Orchestrator Changes

### Pipeline

- `__init__`: rename `function_database` → `result_database`; add optional `status_database` and `log_database` override parameters; store raw user-provided root database
- `run()` gains an optional `observer` parameter
- If `pipeline_database` is set and no `observer` is provided, `Pipeline` creates:
  ```python
  observer = CompositeObserver(
      StatusObserver(self.status_database),
      LoggingObserver(self.log_database),
  )
  ```
- If an explicit `observer` is passed to `run()`, it is used as-is (no default observers created)
- `save(level="full")` serializes observer configs using `to_config()` on each observer
- `load()` reconstructs the scoped database hierarchy from the root database; no old-format backward compat; all `pipeline_path_prefix` hint logic removed

### Orchestrators

- `SyncPipelineOrchestrator.__init__()` loses the `observer` parameter
- `SyncPipelineOrchestrator.run(graph, *, observer=None, ...)` gains it
- Same change for `AsyncPipelineOrchestrator`

## Section 6: Database Serialization

### Scoped database tracking

Databases created via `db.at(*path)` store two additional fields:
- `_root`: reference to the root (non-scoped) database instance
- `_scoped_path`: the full path tuple from root to this view

Root databases have `_root = None` and `_scoped_path = ()`.

### Serialization format

`to_config(db_registry=None)` checks `_root`:
- **Root database**: serializes full connection config as before
- **Scoped database**: serializes as a reference:
  ```json
  {
    "type": "scoped",
    "ref": "<root_config_hash>",
    "path": ["_result"]
  }
  ```
  The root config is registered into `db_registry` under its content hash.

### DatabaseRegistry

`DatabaseRegistry.register(config) -> str` deduplicates by content hash (JSON-stable serialization). Returns the hash key. Scoped databases reference root configs by this key, so the full connection config appears only once in the saved pipeline regardless of how many scoped views are used.

### Deserialization

`resolve_database_from_config(config, registry)`:
- For `"scoped"` type: resolves the referenced root config, then calls `.at(*config["path"])`
- For all other types: existing logic unchanged

This applies to all database implementations: `InMemoryArrowDatabase`, `DeltaTableDatabase`, and `ConnectorArrowDatabase`.

## Section 7: ObservabilityReader Update

`ObservabilityReader._discover_tables()` searches for `"_status"` and `"_log"` path components (replacing the old `"status"` and `"logs"`). All other logic — recursive `_delta_log` discovery, `node_name` extraction, column rename mappings, `_clean_status_df`, `_clean_logs_df` — remains unchanged.

## Data Flow Summary

```
User provides: root_db, pipeline_name
  │
  ▼
Pipeline.compile()
  pipeline_db = root_db.at(*name)
  result_db   = pipeline_db.at("_result")
  status_db   = pipeline_db.at("_status")
  log_db      = pipeline_db.at("_log")
  │
  ├─► FunctionNode.attach_databases(pipeline_db, result_db)
  │     node_identity_path = pod.uri + schema_hash + content_hash
  │     CachedFunctionPod writes to result_db at record_path=pod.uri
  │
  └─► Pipeline.run(observer=None)
        creates CompositeObserver(StatusObserver(status_db), LoggingObserver(log_db))
        passes observer to orchestrator.run(graph, observer=observer)
          │
          └─► for each node:
                ctx_obs = observer.contextualize(*node.node_identity_path)
                ctx_obs.on_node_start(label, hash, tag_schema=...)
                ctx_obs.create_packet_logger(tag, packet)
                  └─► writes to status_db / log_db at identity_path
```
