# ENG-340 + ENG-349: Decouple Nodes from pipeline_path via ScopedDatabase & Observer Storage Path Migration

**Date:** 2026-04-01
**Issues:** ENG-340 (Refactor: decouple nodes from pipeline_path via ScopedDatabase), ENG-349 (Observer/Logger: migrate storage path scoping)
**Branch target:** `dev`

---

## Overview

This refactor removes `pipeline_path` from the node and observer APIs by pre-scoping databases before passing them in. Nodes and observers no longer receive a path prefix â€” they receive databases already rooted at the correct location. The combined scope covers ENG-340 (node decoupling) and ENG-349 (observer storage path migration), since both share the same mechanism and it is cleaner to do them together.

No backward compatibility is maintained for previously saved pipeline formats. This is still greenfield.

> **Current vs. target state**: All sections below describe the *target* API after this refactor. Where a current signature is mentioned for comparison, it is explicitly labeled "current". Discrepancies between this spec and the existing source code are expected and are the intended changes.

---

## Section 1: Database Scoping (Pipeline-level)

`Pipeline.compile()` creates a hierarchy of pre-scoped databases from the user-provided root database:

```
pipeline_db  = user_db.at(*pipeline_name_parts)
result_db    = pipeline_db.at("_result")
status_db    = pipeline_db.at("_status")
log_db       = pipeline_db.at("_log")
```

Each derived database is passed directly to the subsystem that needs it:
- `result_db` â†’ `FunctionNode.attach_databases(pipeline_database, result_database)`
- `status_db` â†’ `StatusObserver(status_db)`
- `log_db` â†’ `LoggingObserver(log_db)`

Nodes and observers write paths relative to their pre-scoped database root. The pipeline name and type partition are invisible to them.

### Path component naming

The sub-path components use underscore prefixes and singular names:
- `"_result"` (new partition, no prior equivalent)
- `"_status"` (replaces old `"status"`)
- `"_log"` (replaces old `"logs"` â€” note: singular, not plural)

### Pipeline properties

`Pipeline` exposes read-only properties:
- `scoped_pipeline_database` â€” the `pipeline_db` scoped view
- `result_database` â€” the `_result`-scoped view
- `status_database` â€” the `_status`-scoped view
- `log_database` â€” the `_log`-scoped view

The constructor parameter is renamed `function_database` â†’ `result_database`. The old `function_database` property is also removed and replaced by `result_database`.

## Section 2: Node Interface Changes

### FunctionNode

- Remove `_pipeline_path_prefix: tuple[str, ...]` field
- Remove `_stored_pipeline_path` and all backward-compat prefix-stripping logic
- Rename `pipeline_path` property â†’ `node_identity_path`; returns `pf.uri + (f"schema:{pipeline_hash}", f"instance:{content_hash}")` relative to the pre-scoped database
- `attach_databases(pipeline_database, result_database=None)` â€” `pipeline_path_prefix` parameter removed entirely
- `execute()` uses `observer.contextualize(*self.node_identity_path)` to bind node identity before calling hooks

#### FunctionNode â†’ CachedFunctionPod wiring

`FunctionNode.attach_databases()` instantiates `CachedFunctionPod` internally. After this refactor, the call becomes:

```python
self._cached_pod = CachedFunctionPod(
    function_pod=self._function_pod,
    result_database=result_database,   # already pre-scoped by pipeline
    # record_path_prefix removed
)
```

`CachedFunctionPod` then creates `ResultCache(result_database=result_database, record_path=self.uri)` â€” no prefix needed because `result_database` is already scoped to `"_result"`.

### OperatorNode

- Same changes: remove `_pipeline_path_prefix`, rename `pipeline_path` â†’ `node_identity_path`
- `attach_databases(pipeline_database, cache_mode=CacheMode.OFF)` â€” no prefix parameter
- `from_descriptor()` removes all `pipeline_path_prefix` handling

### CachedFunctionPod

- Remove `record_path_prefix: tuple[str, ...] = ()` parameter from `__init__`
- `self._cache = ResultCache(result_database=result_database, record_path=self.uri)` â€” path is just `self.uri`, no prefix, since `result_database` is already pre-scoped

## Section 3: Observer Protocol

`ExecutionObserverProtocol` changes (current â†’ target):

| Current | Target |
|---------|--------|
| `contextualize(node_hash, node_label)` | `contextualize(*identity_path: str)` |
| `on_node_start(label, hash, pipeline_path=(), tag_schema=None)` | `on_node_start(label, hash, tag_schema=None)` |
| `on_node_end(label, hash, pipeline_path=())` | `on_node_end(label, hash)` |
| `create_packet_logger(tag, packet, pipeline_path=())` | `create_packet_logger(tag, packet)` |

`contextualize(*identity_path)` returns a context-bound observer (or self). The identity path flows from `node_identity_path` on the node. All storage path resolution is encapsulated inside the observer implementation â€” callers never compute paths.

`NoOpObserver.contextualize(*identity_path)` returns `self` (no state needed). Current implementations in `observer.py`, `composite_observer.py`, and `status_observer.py` still use the two-argument `contextualize(node_hash, node_label)` form â€” these are the target changes.

The `on_run_start` docstring in `observability_protocols.py` currently contains: _"Observers that need node-level storage scoping should use the `pipeline_path` supplied per-node via `on_node_start`"_ â€” this must be rewritten since `pipeline_path` no longer exists on `on_node_start`.

## Section 4: Observer Implementations

### StatusObserver

```python
class StatusObserver:
    def __init__(self, status_database: ArrowDatabaseProtocol) -> None: ...
    def contextualize(self, *identity_path: str) -> _ContextualizedStatusObserver: ...
    def to_config(self) -> dict: ...
    @classmethod
    def from_config(cls, config: dict) -> StatusObserver: ...

class _ContextualizedStatusObserver:
    # Bound to identity_path; writes to status_db.add_record(identity_path, ...)
```

- Receives pre-scoped `status_database` â€” no path mirroring logic
- `contextualize(*identity_path)` returns a lightweight wrapper with the path bound
- `_node_context` dict removed; context is held in the wrapper, not mutated state
- `get_status()` loses the `pipeline_path` parameter

### LoggingObserver

Same pattern:
- Pre-scoped `log_database`
- `contextualize(*identity_path)` â†’ `_ContextualizedLoggingObserver`
- `PacketLogger` receives `log_path=identity_path` directly
- `get_logs()` loses `pipeline_path` parameter

### CompositeObserver

`contextualize(*identity_path)` delegates to each child and wraps results in a new `CompositeObserver`.

The module-level docstring example in `composite_observer.py` currently shows `SyncPipelineOrchestrator(observer=observer)` â€” this must be updated to `orchestrator.run(graph, observer=observer)` to reflect the observer-to-`run()` migration.

## Section 5: Pipeline API and Orchestrator Changes

### Pipeline

- `__init__`: rename `function_database` â†’ `result_database`; add optional `status_database` and `log_database` override parameters; store raw user-provided root database. Remove the old `function_database` property.
- `run()` gains an optional `observer` keyword parameter:
  ```python
  def run(self, *, observer=None, materialize_results=True, run_id=None, pipeline_uri="")
  ```
- If `pipeline_database` is set and no `observer` is provided, `Pipeline` creates:
  ```python
  observer = CompositeObserver(
      StatusObserver(self.status_database),
      LoggingObserver(self.log_database),
  )
  ```
- If `pipeline_database` is **not** set (or is `None`), the default observer is `NoOpObserver()`.
- If an explicit `observer` is passed to `run()`, it is used as-is (no default observers created)
- `Pipeline.run()` passes the observer to the orchestrator: `orchestrator.run(graph, observer=observer, ...)`
- `save(level="full")` serializes observer configs using `to_config()` on each observer
- `load()` reconstructs the scoped database hierarchy from the root database; no old-format backward compat; all `pipeline_path_prefix` hint logic removed

### Orchestrators

`observer` moves from `__init__` to `run()` on both orchestrators:

```python

    ...

    ...

    graph,
```

## Section 6: Database Serialization

> **Note**: The `DatabaseRegistry` class already exists in `serialization.py` with `register(config)`, `resolve(key)`, `to_dict()`, and `from_dict()`. Keys use the format `db_<sha256[:8]>` (e.g. `"db_a1b2c3d4"`). No changes to the class itself are needed â€” the work here is adding `_root`/`_scoped_path` to database instances and updating their `to_config()` methods.

### Scoped database tracking

Databases created via `db.at(*path)` gain two new instance fields (existing `_path_prefix` is kept for functional routing):
- `_root`: reference to the original root database instance (`None` for root databases)
- `_scoped_path`: full path tuple from root to this view (empty `()` for root databases)

This applies to `InMemoryArrowDatabase`, `DeltaTableDatabase`, `ConnectorArrowDatabase`, and `NoOpArrowDatabase`.

### Serialization format

`to_config(db_registry=None)` checks `_root`:
- **Root database** (`_root is None`): serializes full connection config as before; for `InMemoryArrowDatabase` the `base_path` field remains `[]`
- **Scoped database** (`_root is not None`): serializes as a reference, omitting `base_path`:
  ```json
  {
    "type": "scoped",
    "ref": "db_a1b2c3d4",
    "path": ["_result"]
  }
  ```
  The root config is registered via `db_registry.register(self._root.to_config())`.

For `NoOpArrowDatabase`: scoped instances emit the `{"type": "scoped", ...}` format; root instances continue to emit their current no-op config.

When serializing a pipeline, a single `DatabaseRegistry` instance is created and passed through all `to_config(db_registry=registry)` calls. The resulting `registry.to_dict()` is stored under the `"databases"` top-level key in the save file.

Note: `DatabaseRegistry` (the deduplication class) is distinct from `DATABASE_REGISTRY` (the type-string-to-class dict also in `serialization.py`).

### Deserialization

`resolve_database_from_config` gains an optional second parameter (current: one argument):

```python
def resolve_database_from_config(
    config: dict[str, Any],
    registry: dict[str, dict] | None = None,
) -> Any:
```

- `registry` is the `"databases"` dict from the save file (`{key: root_config}`)
- For `"scoped"` type: `resolve_database_from_config(registry[config["ref"]], registry).at(*config["path"])`
- For all other types: existing logic unchanged (no registry needed)

Existing call sites that pass only `config` continue to work since `registry` defaults to `None`.

## Section 7: ObservabilityReader Update

`ObservabilityReader._discover_tables()` searches for `"_status"` and `"_log"` path components. This replaces the current `"status"` (â†’ `"_status"`) and `"logs"` (â†’ `"_log"`) â€” note both the underscore prefix addition and the plural-to-singular rename on the log component.

**Node name extraction**: the `node_identity_path` has multiple components: `(pod_label, f"schema:{hash}", f"instance:{hash}")`. Status/log Delta tables are written at the full `identity_path` within their pre-scoped database, so the on-disk path under `_status` looks like `../_status/<pod_label>/schema:<hash>/instance:<hash>/_delta_log`. The existing `parts[idx + 1]` extraction correctly yields `pod_label` (the first component of the identity path and the human-readable node name). No change needed to the extraction logic.

All other logic â€” recursive `_delta_log` discovery, column rename mappings, `_clean_status_df`, `_clean_logs_df` â€” remains unchanged.
